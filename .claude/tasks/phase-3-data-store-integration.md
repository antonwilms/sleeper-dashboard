# Task: Phase 3 — Data Store Integration

## Context

Phase 1 added an IndexedDB export button. Phase 2 stood up the public `sleeper-dashboard-data` repo, served via jsDelivr:

```
https://cdn.jsdelivr.net/gh/<user>/sleeper-dashboard-data@main/<path>
```

Repo layout:
```
nfl/season-totals/<year>.json
college/receiving/<year>.json
college/rushing/<year>.json
college/passing/<year>.json     (planned)
ktc/snapshot-<date>.json
manifest.json                   ({ schemaVersion, lastModified, inProgress } per file)
```

A fresh install today triggers `loadCareerHistory` (18 weeks × ~14 historical seasons, 200ms delay each = ~7 minutes) plus 24 CFBD round-trips (8 years × 3 categories). Phase 3 makes both come from the data store in seconds. Live APIs are reserved for the current/in-progress season and for anything the data store doesn't have yet.

## Decisions

### 1. Module structure — single `src/api/dataStore.js`

**Recommended:** one new module, not per-API logic.

Rationale:
- Manifest fetch/cache, URL construction, schema-version checks, jsDelivr-failure handling, and the `VITE_DATA_STORE_ENABLED` gate are all cross-cutting. Duplicating them in `sleeperStats.js`, `cfbd.js`, `ktc.js` would scatter the pattern and invite drift.
- The existing modules' public APIs (`getSeasonTotals`, `getBulkPlayerStats`, `getKTCValues`) stay intact — they remain the call sites the rest of the app uses. Internally, each one gains a single new line: try `tryDataStore(...)` between the IndexedDB cache check and the live fetch.
- Keeps `dataStore.js` testable in isolation later if we add unit tests.

### 2. Fetch order — Cache → Data store → API

```
1. IndexedDB cache
       │  hit (not expired, not stale-shape) → return
       ▼  miss / expired / stale
2. Data store (if enabled and reachable, file present, schema OK, not inProgress)
       │  hit → write to IndexedDB cache, return
       ▼  miss / inProgress / disabled / schema mismatch / network fail
3. Live API (existing behaviour)
       │  hit → write to IndexedDB cache, return
       ▼  fail → bubble up the original error
```

`inProgress: true` in the manifest means "this file is being regenerated by the data-store CI job; the bytes on disk are not trustworthy right now." Treat it the same as a 404 — fall through to the live API for this request. Do **not** cache `inProgress` responses; the next request can retry the data store.

### 3. Manifest handling

- **Fetched once per app session.** On first call to any `tryDataStore(...)` function, load the manifest and memoise the promise. Subsequent calls reuse it.
- **Persisted to IndexedDB** with a 60-minute TTL under key `data-store/manifest`. Survives reloads within the hour so we don't hammer jsDelivr on every page refresh.
- **Update detection:** when a cache entry for, say, `season-totals/2023` is read, we compare its stored `lastModified` (written alongside the data) against the manifest's current `lastModified` for that file. If the manifest is newer, the cache entry is considered stale — we fall through to the data store again to refresh.
  - This requires a small extension to `setCache` / cache record shape — see "Data shapes" below.
- If the manifest fetch fails (timeout, 404, parse error), `dataStore` enters a "session-disabled" state. All `tryDataStore(...)` calls return null for the rest of the session. Recovery on next reload.

### 4. Failure modes

| Failure | Behaviour |
|---|---|
| jsDelivr times out (> 5 s for manifest, > 15 s for a data file) | Treat as miss; fall through to live API. Log a single `[dataStore] timeout — falling back` per session. |
| Manifest malformed (JSON parse, missing required keys) | Disable data store for the rest of the session; log error once. |
| Specific file 404 | Treat as not in data store; fall through silently (the manifest may be ahead of the deployed tree). |
| `schemaVersion` in manifest exceeds the app's known max | Skip that file; fall through. Log once per file per session. |
| File present but JSON shape doesn't match expected (validator fails) | Skip; fall through. Log a `[dataStore] shape mismatch <file>` so we notice schema drift quickly. |
| jsDelivr returns 5xx | Same as timeout — single log, fall through. |

Each file the data store can serve gets a tiny shape validator (`isValidSeasonTotals`, `isValidCFBDRows`) — a 3-line function that checks a couple of expected keys on a sample entry. Cheap insurance against silent corruption.

### 5. Configuration

Two env vars read once at `dataStore.js` module load:

| Env var | Default | Purpose |
|---|---|---|
| `VITE_DATA_STORE_URL` | `https://cdn.jsdelivr.net/gh/<user>/sleeper-dashboard-data@main` | Base URL. Override for staging or local file serving during dev. |
| `VITE_DATA_STORE_ENABLED` | `'true'` | Set to `'false'` to force API-only mode for testing. Read as a string; only the literal `'false'` disables. |

Both go in `.env.local`. README gains a "Data store" section listing both.

**No runtime UI toggle.** The debug panel already has cache-clearing buttons; we add one more — "Clear data store cache" — that calls `clearCache('data-store/')`. That's enough manual control for now.

### 6. Scope of phase 3 — NFL season totals + CFBD; skip KTC

Wire up two of the three loaders:

- ✅ **NFL season totals** (`getSeasonTotals` in `sleeperStats.js`). Biggest win — eliminates the ~7-minute career history loader for historical seasons.
- ✅ **CFBD receiving / rushing / passing** (`getBulkPlayerStats` in `cfbd.js`). Second-biggest win — 24 sequential round-trips become 24 jsDelivr edge hits in parallel.
- ⏸️ **KTC** — defer. KTC snapshots are date-stamped (`snapshot-<date>.json`) so the lookup is "find latest" rather than a deterministic path, which is a different access pattern. KTC is already cached aggressively (3 days) and isn't a fresh-install bottleneck. Phase 4 problem.

This keeps the phase 3 surface area focused on the two loaders that actually hurt first-run UX, while validating the architecture against two different schema shapes (season totals map vs CFBD row array).

## New files to create

### `src/api/dataStore.js`

Single module wrapping all data store access.

**Exports:**

```js
// Returns parsed JSON for `relativePath` from the data store, or null if not available.
// Validates schema version and inProgress flag against the manifest before returning.
async function tryDataStore(relativePath, { validate = null } = {})

// True if the data store is enabled and the manifest loaded successfully.
async function isDataStoreReady()

// Returns the manifest entry for a path, or null. Used by callers that need
// lastModified for stale-cache detection.
async function getManifestEntry(relativePath)

// Force re-fetch of the manifest. Called from the debug panel.
function invalidateManifest()
```

**Module-private state:**

- `enabled` (bool, from env)
- `baseUrl` (string, from env)
- `manifestPromise` (Promise<Manifest> | null, memoised per session)
- `sessionDisabled` (bool, set true on manifest failure)
- `loggedKeys` (Set, for de-duplicating warnings)

## Data shapes

### Manifest (`manifest.json`)

```js
{
  schemaVersion: 1,                       // overall manifest schema
  generatedAt: '2026-05-17T00:00:00Z',
  files: {
    'nfl/season-totals/2023.json':        { schemaVersion: 1, lastModified: '2026-04-01T...', inProgress: false },
    'college/receiving/2023.json':        { schemaVersion: 1, lastModified: '2026-04-01T...', inProgress: false },
    'college/rushing/2023.json':          { schemaVersion: 1, lastModified: '2026-04-01T...', inProgress: false },
    'college/passing/2023.json':          { schemaVersion: 1, lastModified: '2026-04-01T...', inProgress: false },
    // ...
  },
}
```

Manifest is fetched as plain JSON via `fetch`, parsed once, memoised.

### Extended cache record

```js
// Existing record (unchanged for callers that don't need meta)
{ key, data, expiresAt }

// New optional fields written by setCacheWithMeta
{ key, data, expiresAt, sourceLastModified?, sourceSchemaVersion? }
```

`getCache(key) → data` keeps its current behaviour. New helper:

```js
// Returns the full record (including meta), or null.
async function getCacheRecord(key) → { data, expiresAt, sourceLastModified?, sourceSchemaVersion? } | null
```

Only the data-store-aware callers use `getCacheRecord`; they use `sourceLastModified` to decide whether to short-circuit on the manifest stale-check.

### Validator signature

```js
function isValidSeasonTotals(parsed) → boolean   // checks parsed is object, sample value has gamesPlayed/fantasyPoints/dnpWeeks
function isValidCFBDRows(parsed)     → boolean   // checks parsed is array, sample row has playerId/statType/stat
```

## Fetch-order pseudocode (representative — `getSeasonTotals`)

```js
async function getSeasonTotals(season, activePlayerIds, scoringSettings, playersMap, onWeekProgress) {
  const cacheKey = `season-totals/${season}`

  // (1) Read cache record (including meta)
  const record = await getCacheRecord(cacheKey)
  if (record && record.data && Object.keys(record.data).length > 0) {
    const sample = Object.values(record.data)[0]
    if (sample.dnpWeeks !== undefined) {
      // Manifest stale check — only if we have a recorded source
      if (record.sourceLastModified) {
        const entry = await getManifestEntry(`nfl/season-totals/${season}.json`)
        if (entry && entry.lastModified > record.sourceLastModified) {
          // Manifest has a newer version — fall through and re-pull from data store
        } else {
          onWeekProgress?.(18, true)
          return record.data
        }
      } else {
        onWeekProgress?.(18, true)
        return record.data
      }
    }
    // else: pre-gp-fix cache, fall through
  }

  // (2) Try data store
  const dsPath = `nfl/season-totals/${season}.json`
  const dsResult = await tryDataStore(dsPath, { validate: isValidSeasonTotals })
  if (dsResult !== null) {
    const entry = await getManifestEntry(dsPath)
    await setCacheWithMeta(cacheKey, dsResult, 999999, {
      sourceLastModified: entry?.lastModified ?? null,
      sourceSchemaVersion: entry?.schemaVersion ?? null,
    })
    onWeekProgress?.(18, true)
    console.log(`[career] ${season}: loaded from data store (${Object.keys(dsResult).length} players)`)
    return dsResult
  }

  // (3) Live API — existing 18-week loop, unchanged
  // ...
}
```

`tryDataStore` itself, sketched:

```js
async function tryDataStore(relativePath, { validate } = {}) {
  if (!enabled || sessionDisabled) return null
  const manifest = await loadManifest()  // memoised
  if (!manifest) return null

  const entry = manifest.files[relativePath]
  if (!entry) return null
  if (entry.inProgress) return null
  if (entry.schemaVersion > MAX_SUPPORTED_SCHEMA) {
    logOnce(`schema-too-new:${relativePath}`, ...)
    return null
  }

  try {
    const res = await fetchWithTimeout(`${baseUrl}/${relativePath}`, 15000)
    if (!res.ok) return null
    const json = await res.json()
    if (validate && !validate(json)) {
      logOnce(`shape-mismatch:${relativePath}`, ...)
      return null
    }
    return json
  } catch (err) {
    logOnce(`fetch-fail:${relativePath}`, err.message)
    return null
  }
}
```

## Integration points

- **No changes** to `App.jsx` data flow, the playerRows pipeline, `usePlayerProfile`, `ProfileDataContext`, or any component. The data store layer is invisible above the `getSeasonTotals` / `getBulkPlayerStats` boundary.
- **No changes** to `loadCareerHistory` or `loadCollegeStats` — they call the wrapped functions and stay oblivious.
- **One UI change**: the career progress bar will tick through historical seasons very quickly (one tick per season instead of 18 per season). That's a visible UX win, not a bug.

## Acceptance criteria

- [ ] Fresh install (clear all cache, hard reload): `loadCareerHistory` for seasons 2012–N completes in **under 15 seconds** total (was ~7 minutes), with each season showing in the progress bar as a single tick rather than 18 weekly ticks.
- [ ] Network panel shows one request per historical season hitting `cdn.jsdelivr.net`, plus one request for `manifest.json`. **Zero** requests to `api.sleeper.com/stats/nfl/<year>/<week>` for historical seasons. Current-season requests (if any) still go to Sleeper.
- [ ] CFBD: 24 requests to jsDelivr (3 categories × 8 years), **zero** to `api.collegefootballdata.com` for historical years.

- [ ] Manifest stale-check: manually edit a cached record's `sourceLastModified` in IndexedDB DevTools to a past timestamp, reload — the data store is re-pulled for that file.
- [ ] Simulate jsDelivr 404 (point `VITE_DATA_STORE_URL` at a bogus domain): app falls back to live API silently, console shows a single `[dataStore] timeout — falling back` log per session, no errors thrown to UI.
- [ ] Debug panel "Clear data store cache" button removes all `data-store/*` keys and forces a manifest re-fetch on the next request.
- [ ] `npm run build` passes with no new warnings.

## Out of scope

- **KTC integration** — phase 4.
- **Player map (`getAllPlayers`)** from Sleeper — not in the data store yet. Still hits the live API.
- **Weekly stats / projections** for the current season — by design, live only.
- **Write path to the data store** — phase 3 is read-only. The export button from phase 1 is still the only write path, and a separate CI job (out of scope here) ingests those exports.
- **HTTP cache headers / `If-Modified-Since`** — relying on jsDelivr's edge cache + our manifest stale-check. ETag-based revalidation is a future optimisation.
- **Refactoring `loadCareerHistory` / `loadCollegeStats`** — they remain orchestrators; the data-store layer slots in below them.

## Open questions

1. **Manifest path** — fixed at `<baseUrl>/manifest.json`, or configurable? Recommend: fixed for now. If we ever serve from a CDN path that doesn't have the manifest at the root, we add `VITE_DATA_STORE_MANIFEST` then.
2. **Schema version compatibility window** — pin `MAX_SUPPORTED_SCHEMA = 1` in `dataStore.js`. When schema bumps, bump this constant deliberately. Backward compatibility (older app, newer data) is handled by skipping unsupported files.
3. **CDN choice** — jsDelivr is fine. If we hit rate limits or 50 MB file caps later, switch to GitHub Pages or R2. The `VITE_DATA_STORE_URL` env var makes this a one-line change.
4. **Progress bar UX** — the career loader currently animates 18 weeks per season. With the data store, that becomes "season loaded" in one tick. Worth a copy tweak ("Loading 2023 from data store…")? Defer to feel during implementation.
