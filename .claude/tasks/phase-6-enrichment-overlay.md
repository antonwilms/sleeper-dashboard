# Task: Phase 6 — Enrichment Overlay

## Context

Phases 1–5 built the longitudinal data store: primary data is pulled from external APIs (Sleeper, CFBD, KTC) by deterministic scripts and lives in `sleeper-dashboard-data`. Phase 5 most recently added per-week `weeklyStatus` arrays and `availability` aggregates to season totals.

Phase 6 adds an **enrichment overlay**: a layer for hand-curated data no API provides — coaching changes, scheme shifts, injury type/severity for known absence segments, and free-form notes. The data is sparse (entry-per-event, not per player-week) and grows slowly. The mechanism matters more than the volume.

## Goal

Stand up a separate `enrichment/` directory in the data repo with four typed sub-schemas (coaching, scheme, injuries, notes), a `bin/enrich.mjs` CLI to add and validate entries, a read-only enrichment loader in the app that merges overlay data into the player profile at view time, and one minimal UI surface (injury type/severity rendered as a hover-state on `AvailabilityHistory`'s red `D` cells). All other surfaces are deferred — the data accumulates first, the consumers come later.

---

## Decisions

### 1. File structure — **per-type aggregate files, year-keyed inside; one shared registry per type**

Recommended layout under `sleeper-dashboard-data/`:

```
enrichment/
  coaching.json       — all coaching staffs/changes, all years, all teams
  scheme.json         — all scheme assignments, all years, all teams
  injuries.json       — all injury enrichments, keyed by playerId
  notes.json          — all free-form notes (per-player and per-team)
  README.md           — short doc on how to add entries
```

Rationale:
- **One file per type** keeps each schema self-contained and small (an entire season of NFL coaching staffs is ~32 rows × ~3 fields × 14 years ≈ <100 KB even with a permissive shape). Splitting per-year creates 14× the files without making any of them more readable.
- **Not per-team subdirs**: the join key for the app is usually "look up team X in year Y" or "look up player P". A flat per-type file lets a single fetch answer either. Per-team dirs would mean either fetching 32 files or maintaining an index.
- **Sparse registry rejected (option d)**: a single (year, team, player_id) registry collapses four distinct shapes into a hash, hiding the schema. We want to be able to open `coaching.json` and read the file end-to-end.
- Edit ergonomics: each file is small enough that I'll routinely open it in an editor. JSON, pretty-printed, sorted by year then team (or playerId). Merge conflicts are rare and localised to the row I touched.
- App fetch: one fetch per type per session (4 small files), cached with the usual TTL.

### 2. Schema for each enrichment type

All four files share a top-level wrapper:

```json
{
  "schemaVersion": 1,
  "updatedAt": "2026-05-19T12:00:00Z",
  "entries": [ … ]
}
```

`entries` is an array; each entry has a stable `id` (string, generated by the CLI: `<type>-<year>-<team-or-player>-<short-hash>`) so future edits can target a row deterministically. Unknown keys are tolerated on read — additive forward-compatibility.

#### 2a. Coaching (`coaching.json`)

One entry **per (team, year, role)** triple. `role ∈ {"HC", "OC", "DC"}`.

```json
{
  "id":         "coach-2024-SF-HC-1a2b",
  "year":       2024,
  "team":       "SF",
  "role":       "HC",
  "name":       "Kyle Shanahan",
  "tenureStart": 2017,
  "isNew":      false,
  "predecessor": null,
  "source":     "team site, 2024-01-15",
  "notes":      ""
}
```

Required: `id`, `year`, `team`, `role`, `name`. Optional: `tenureStart`, `isNew`, `predecessor`, `source`, `notes`.

#### 2b. Scheme (`scheme.json`)

One entry per (team, year). Captures the dominant offensive/defensive philosophy as I understand it.

```json
{
  "id":             "scheme-2024-MIA-4c8d",
  "year":           2024,
  "team":           "MIA",
  "offense":        "wide zone / play-action",
  "defense":        "vic-fangio-tree zone",
  "tempo":          "fast",
  "changedFromPrev": false,
  "source":         "PFF preview 2024",
  "notes":          ""
}
```

Required: `id`, `year`, `team`. At least one of `offense` / `defense` / `tempo` should be set. Optional everything else. Free-form strings — no enum, by design (the taxonomy will drift).

#### 2c. Injuries (`injuries.json`)

One entry per **known injury event**, referenced to an absence segment in `season-totals`. The CLI validator confirms the segment exists.

```json
{
  "id":         "inj-6803-2023-w2-9e1f",
  "playerId":   "6803",
  "year":       2023,
  "segmentStartWeek": 2,
  "segmentEndWeek":   18,
  "type":       "ACL",
  "bodyPart":   "knee",
  "severity":   "season-ending",
  "dateInjured": "2023-09-11",
  "dateReturned": null,
  "source":     "team announcement 2023-09-11",
  "notes":      "non-contact, Week 1 Day 1 of season"
}
```

Required: `id`, `playerId`, `year`, `segmentStartWeek`. `segmentEndWeek` defaults to `null` (use the season-totals derived end). `type`/`bodyPart`/`severity`/`dateInjured`/`dateReturned`/`source`/`notes` all optional. `severity ∈ {"season-ending", "multi-week", "single-game", "playing-through"}` is the *suggested* enum but is not validated as a closed set (forward-compat).

#### 2d. Notes (`notes.json`)

Catch-all. Each entry is scoped to either a `playerId` or a `team` (exactly one of the two), with optional `year`.

```json
{
  "id":       "note-4034-2024-2f3a",
  "playerId": "4034",
  "team":     null,
  "year":     2024,
  "tag":      "usage",
  "body":     "Slot-heavy alignment in 11-personnel through Week 6; PFF route data.",
  "source":   "PFF article, 2024-10-22"
}
```

Required: `id`, `body`, exactly one of `playerId`/`team`. Optional: `year`, `tag`, `source`.

### 3. How enrichment merges with primary data — **(a) fetched alongside; merged in-memory at view time; no change to primary files**

Recommended:
- Primary data files (`nfl/season-totals/*.json`) remain untouched. They're produced by deterministic API scripts and re-running them must not be lossy.
- Enrichment is fetched as a separate group of 4 small files via `tryDataStore('enrichment/<file>.json')` on app boot (in parallel with manifest and career history loading).
- The 4 files are exposed through a new `enrichmentMap` value in `ProfileDataContext` (and read into a top-level `App.jsx` state, mirroring `careerStats`).
- Merging happens at the consumer (right now, just `AvailabilityHistory`'s tooltip). Consumers look up enrichment by key (playerId / team / year) on render. No bake step.

Rationale:
- **Option (b) bake-into-totals is rejected**: it couples a slow regen of primary files to a small text edit and makes enrichment-only changes invisible to anyone not running the pipeline. Worst of both worlds.
- **Option (c) lazy on profile-open is rejected for now**: the 4 enrichment files together are small (probably < 200 KB combined within a few years). Fetching upfront makes the rest of the app's lookups synchronous and trivial. Revisit if files grow past ~500 KB each.
- **Option (a)** wins on simplicity: enrichment is its own concern, has its own update path, and the app's primary data path stays exactly as phase 5 left it.

When enrichment is updated but season-totals aren't: the app sees the new enrichment on the next session (or next manifest TTL expiry, 60 min). Conversely if season-totals are regenerated and the segment endpoints shift, an injury entry might become "orphaned" — the validator catches this on the *next* CLI run (see decision 6); the app silently ignores orphaned entries.

### 4. Where enrichment surfaces — **one surface in phase 6: injury type/severity on `AvailabilityHistory`'s D-cells**

Phase 6 ships exactly one consumer:

- **`AvailabilityHistory` sparkline tooltip**: when hovering a `D` cell, if an injury enrichment entry's `segmentStartWeek..segmentEndWeek` covers that week for this `(playerId, year)`, the tooltip shows the type/severity inline. Example: `W4: DNP — ACL (season-ending)`. Cells with no matching enrichment show the existing `W4: DNP`. Same Tooltip component; only the string changes.

Deferred (phase 7+):
- Coaching/scheme info in a team panel or in player profile's team context block.
- A dedicated "Team history" tab.
- Notes display anywhere.
- Any consumption by `dynastyScore.js`.

Rationale: until I've actually populated entries across multiple seasons, building UI for them is speculative. One injury-type surface is enough to validate the mechanism end-to-end and gives an immediate win for the absence segments already in the app.

### 5. Entry workflow — **(b) CLI in the data repo**

Recommended: `node bin/enrich.mjs <type> [args…]` in `sleeper-dashboard-data`.

Why CLI over manual edits or in-app UI:
- **Validation at write time** (decision 6) — manual edits can produce broken JSON that the app silently drops; a CLI fails loudly.
- **ID generation** is deterministic and the human doesn't have to think about it.
- **Predictable formatting** — the CLI uses the same `writeJsonStable` writer as the existing scripts, so diffs are minimal.
- **In-app editor** is over-engineered for entry cadence measured in weeks.
- I'll still occasionally hand-edit (e.g. typo fix). The validator can be run standalone as `node bin/enrich.mjs validate` to catch issues from manual edits before committing.

CLI shape:

```bash
# Coaching
node bin/enrich.mjs coaching add --year 2024 --team SF --role HC --name "Kyle Shanahan" \
    [--tenure-start 2017] [--is-new] [--predecessor "Chip Kelly"] [--source "..."] [--notes "..."]

# Scheme
node bin/enrich.mjs scheme add --year 2024 --team MIA \
    [--offense "wide zone"] [--defense "..."] [--tempo fast] [--changed-from-prev] [--source "..."]

# Injuries (segment must exist in season-totals for that playerId/year)
node bin/enrich.mjs injuries add --player 6803 --year 2023 --segment-start 2 \
    [--type ACL] [--body-part knee] [--severity season-ending] \
    [--date-injured 2023-09-11] [--date-returned 2023-12-15] [--source "..."]

# Notes
node bin/enrich.mjs notes add --player 4034 --year 2024 \
    --body "Slot-heavy alignment in 11-personnel through Week 6." [--tag usage] [--source "..."]
# or:
node bin/enrich.mjs notes add --team SF --year 2024 --body "..." [--tag scheme]

# Maintenance
node bin/enrich.mjs validate                # validate all four files
node bin/enrich.mjs list <type> [--year Y]  # print current entries
node bin/enrich.mjs remove <id>             # remove by id (any type)
```

ID format: `<type-prefix>-<year>-<key>-<6 hex>` where `key` is `team` for coaching/scheme, `playerId` for injuries/notes(playerId), `team` for notes(team). The 6-hex suffix is `sha1(JSON.stringify(rest))[0..6]` so the same fields always produce the same id (idempotent `add`).

Behaviour: `add` is upsert by id — re-running with identical fields is a no-op; running with same key but different fields prints a diff and refuses without `--force`.

### 6. Validation — **strict at write time (CLI), lenient at read time (app)**

#### At write time (CLI)

- **All four files**: top-level shape (`schemaVersion`, `entries[]`); every entry has its required fields per decision 2; no duplicate `id`s.
- **Coaching**: `team` is a valid NFL abbreviation (closed list from a constant); `year ∈ [2012, currentSeason+1]`; `role ∈ {HC, OC, DC}`; only one entry per (year, team, role).
- **Scheme**: `team` valid; `year` in range; at least one of `offense`/`defense`/`tempo` is non-empty; one entry per (year, team).
- **Injuries**: `playerId` resolves against `raw/-players-nfl.json` (the player map already in the data repo); `year` in range; `(playerId, year)` exists in `nfl/season-totals/<year>.json`; the player's `availability.absenceSegments` in that file contains a segment whose `start === segmentStartWeek` (i.e. you can't enrich an injury that didn't happen per the data).
- **Notes**: exactly one of `playerId`/`team` set; if `playerId`, resolves against player map; if `team`, valid abbreviation; `year` if present in range; `body` non-empty.

The validator runs:
- Implicitly on every `add` / `remove` (the modified file is re-validated before write).
- Explicitly on `node bin/enrich.mjs validate`.
- In CI, as part of the existing GitHub Action smoke step (one extra line).

#### At read time (app)

- `tryDataStore('enrichment/<file>.json')` only checks top-level shape (`schemaVersion`, `entries` is an array). If shape fails, the app continues without that enrichment type — never blocks.
- Per-entry: the consumer (e.g. the tooltip resolver) treats missing/unknown fields as absent. No throws on malformed entries.
- Orphaned injury entries (segment no longer exists because season-totals was regenerated): silently ignored. The CLI's `validate` flags them on next run.

### 7. Bootstrapping — **(b) seed current-season coaching staffs only**

Ship phase 6 with:
- Empty `scheme.json`, `injuries.json`, `notes.json` (just the wrapper + empty `entries`).
- `coaching.json` containing **2025** HC/OC/DC for all 32 teams (96 entries). That exercises the schema end-to-end and gives the validator real data to chew on the first time CI runs. I have this information from public sources and can supply it during the implementation session, or the sonnet session can include a TODO marker and ask me to fill it in. **Decision: implementation session writes the schema scaffolding + an empty `coaching.json`. I (the human) populate the 2025 staffs in a follow-up commit using the CLI.** That keeps the implementation session deterministic and removes a "Claude generates a roster" failure mode.

Why not (c) a 3–5 sample: too small to be useful, too easy to forget to clean up. Empty + one CLI session for me afterwards is cleaner.

## App-side fetch and merge (pseudocode)

```js
// On App mount, parallel with manifest + career load:
useEffect(() => {
  (async () => {
    const enrichmentMap = await loadEnrichment()
    setEnrichmentMap(enrichmentMap)  // null entries per type if anything fails
  })()
}, [])

// loadEnrichment:
async function loadEnrichment() {
  const [coaching, scheme, injuries, notes] = await Promise.all([
    tryDataStore('enrichment/coaching.json', { validate: isValidEnrichment }),
    tryDataStore('enrichment/scheme.json',   { validate: isValidEnrichment }),
    tryDataStore('enrichment/injuries.json', { validate: isValidEnrichment }),
    tryDataStore('enrichment/notes.json',    { validate: isValidEnrichment }),
  ])
  return { coaching, scheme, injuries, notes }
}

function isValidEnrichment(payload) {
  return payload && typeof payload === 'object'
    && Array.isArray(payload.entries)
    && Number.isInteger(payload.schemaVersion)
}

// In AvailabilityHistory's Sparkline:
function tooltipFor(code, week) {
  if (code !== 'D') return `W${week}: ${STATUS_LABEL[code]}`
  const hit = findInjuryForWeek(enrichmentMap?.injuries, playerId, season, week)
  if (!hit) return `W${week}: DNP`
  const sev = hit.severity ? ` (${hit.severity})` : ''
  const typ = hit.type ?? 'injury'
  return `W${week}: DNP — ${typ}${sev}`
}
```

`enrichmentMap` is null until the fetch resolves; consumers treat null as "no enrichment available" and render baseline behaviour.

## CLI design (recap)

`bin/enrich.mjs <type> <action> [opts]` where:
- `type ∈ {coaching, scheme, injuries, notes}` for `add` / `list`.
- `action ∈ {add, remove, list, validate}`. `remove` and `validate` are typeless (operate on the id / all files).
- Common flags: `--dry-run` (validate + show diff, no write), `--force` (overwrite an existing id with different fields), `--source`, `--notes`.
- Examples in decision 5 above.

Exit codes: 0 on success, 1 on validation failure, 2 on usage error. Mirrors `bin/update.mjs`.

## Integration points

- **App state**: new `enrichmentMap` state in `App.jsx`, sibling of `careerStats` and `ktcMap`. Loaded once on mount; no re-fetch (manifest TTL handles freshness on next session).
- **Context**: extended `ProfileDataContext` value (additive).
- **Consumers**: only `AvailabilityHistory` in phase 6. Other components see the new context value but ignore it.
- **Pipeline**: `playerRows` pipeline is **untouched**. Enrichment is profile-time, not pipeline-time, in phase 6.
- **Caching**: the four enrichment files ride the existing `tryDataStore` → IndexedDB cache path with default TTL. No bespoke caching.
